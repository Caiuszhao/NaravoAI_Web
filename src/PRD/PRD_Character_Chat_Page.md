# 角色聊天页 PRD

## 1. 文档信息

- 产品名称：NaravoAI Website
- 模块名称：Character Chat Page
- 文档版本：v0.1
- 更新时间：2026-04-26
- 文档性质：产品需求与交互设计草案

## 2. 背景

NaravoAI 的核心不是一个“聊天机器人壳子”，也不是一个“只能走剧情分支的视频产品”。

它真正特别的地方，是把三件通常分散的体验放在了一起：

- 用户先在具体故事里与角色相遇。
- 用户通过文字、语音、动作等方式影响角色当下反应。
- 这些互动不会被剧情页吃掉，而会继续沉淀为角色关系、情绪状态和后续互动语境。

这意味着，用户和角色之间的关系不应该只存在于某一个 `Game`、某一个 `Chapter`、某一个剧情高潮节点里。关系要能在剧情结束后继续存在，也要能在下一次进入剧情前继续预热。

角色聊天页就是为这个目的存在的。

它不是普通 IM 对话页，也不是角色详情页的附属弹窗。它应该被定义为：

**一个承接角色持续关系的核心互动页面，让用户在不进入具体剧情场景的前提下，仍然能与角色进行连续、带上下文的文字或语音对话。**

## 3. 与现有产品结构的关系

当前聊天页需要承接至少三类入口：

1. `Home` 页面的 `cast` 列表中的 `chat with` 按钮。
2. `character profile` 页面 Hero 区的 `chat` 入口。
3. `character profile` 页面 Quick Actions / 底部 CTA 中的 `chat` 入口。

这三个入口虽然都进入同一个页面，但用户心态并不一样：

- 从 `Home` 的 `cast` 列表进入时，用户更像是在“挑一个现在想聊的人”。
- 从 `character profile` 顶部 Hero 进入时，用户通常刚确认完关系状态，想立刻试探一下角色现在的回应。
- 从 `character profile` 下方行动区进入时，用户通常已经浏览过关系脉络、关键节点和相关内容，更像是在“带着明确理解进入对话”。

因此，聊天页本身应统一，但入口上下文要有轻微差异化承接：

- 从 `Home` 进入：强调“快速开始对话”。
- 从 `character profile` 进入：强调“承接当前关系状态与最近互动结果”。

## 4. 产品定位

角色聊天页的定位可以定义为：

**一个介于“剧情互动现场”和“角色关系主页”之间的长期互动容器。它负责承接用户对角色的即时表达、轻量陪伴、关系确认、情绪试探与日常维系。**

它对产品有四层价值：

- 对用户：降低进入互动的门槛，不必每次都回到高成本剧情节点。
- 对角色：强化“TA 是持续存在的人，而不是某段视频里的 NPC”。
- 对关系：把剧情中的高情绪节点，延展成日常可积累的关系过程。
- 对产品：增加非剧情时段的活跃、复访与情感留存。

## 5. 市场参照与机会判断

市场上相近产品大致分成两类，但都没有完整覆盖 NaravoAI 当前要做的事情。

### 5.1 AI companion 类产品给用户建立的预期

这类产品通常已经让用户形成了几个固定预期：

- 角色应该随时可聊。
- 角色应该记得我。
- 文字和语音都应该是自然入口。
- 关系状态会影响对话氛围。

从公开产品信息看，`Replika` 强调长期陪伴、记忆、关系状态、语音/视频通话等能力；`Character.AI` 强调角色丰富度、1:1 对话、语音与 Voice 绑定能力。这类产品的优点是“对话随时开始”，但短板也很明显：

- 对话通常脱离具体叙事上下文。
- 关系变化更多靠体感，不容易看出“为什么会变成现在这样”。
- 角色很多，但角色和用户之间的“共同经历”往往不够强。

### 5.2 互动叙事类产品给用户建立的预期

这类产品更擅长告诉用户：

- 我在什么故事里认识这个角色。
- 我做了什么选择。
- 接下来还能推进哪一章。

以 `Episode` 这类产品为代表，叙事推进和章节驱动很清楚，但角色关系通常还是依附在剧情节点中，缺少一个真正独立的、可以持续对话的关系空间。

### 5.3 NaravoAI 的机会

NaravoAI 最值得抓住的，不是简单复制 companion 产品的聊天页，而是做一个更有“关系证据”的聊天页：

- 用户不是和一个纯设定角色聊天，而是在和“曾经一起经历过具体事件的角色”聊天。
- 对话不是剧情的替代品，而是剧情之外的延续层。
- 角色回应不该只是维持人设，还应该承接最近剧情结果、关系波动和长期记忆。

所以这个页面的价值，不在“能不能发消息”，而在：

- 这个对话为什么现在值得发生。
- 角色为什么会用这种状态回应我。
- 这段对话怎样影响我下一次回到故事时的心理预期和行动选择。

## 6. 角色聊天页满足的用户需求

### 6.1 用户此刻真正想完成的任务

当用户点进聊天页时，通常不是为了机械地发一句话，而是在完成以下任务：

1. 确认这个角色现在还“在”。
2. 试探角色当前情绪和关系温度。
3. 在不进入剧情的情况下继续维持联系。
4. 用一句话或一段语音，表达自己此刻想说的内容。
5. 在进入下一段高情绪剧情前，先做一次低压力预热。
6. 在剧情结束后，继续延长刚建立起来的情绪余波。
7. 感受到产品真的记得他们共同发生过什么。

### 6.2 核心需求拆解

#### 需求一：我要快速开聊，不想先做复杂选择

用户从 `chat with` 入口进入时，目标通常非常直接：现在就想和这个角色说话。

因此聊天页不能让用户先经历一连串阻塞动作，例如：

- 先选模式再进页。
- 先做复杂设定再开聊。
- 先看大量说明再输入。

正确的体验应该是：**进入即能看懂，停留即能开聊。**

#### 需求二：我希望角色知道“我是谁”和“我们刚发生了什么”

如果用户刚结束某个 `Chapter`，然后进入聊天页，第一句回应不能像全新会话。

用户预期角色至少知道：

- 你们是谁。
- 当前关系阶段是什么。
- 最近一次互动发生在哪个故事节点。
- 最近关系是升温、受伤、紧张、还是确认。

如果没有这些承接，聊天页就会把 NaravoAI 最重要的“持续记忆”优势直接抹平。

#### 需求三：我需要文字和语音都成立，而且它们不是两套割裂体验

既然用户可以与角色进行文字或语音对话，那么这两种输入方式必须共享同一段关系上下文，而不是两个互不关联的模式。

用户会期待：

- 我可以直接打字。
- 我也可以按住说话。
- 我说的话被识别后，角色仍然是在同一段对话里回应我。
- 文字与语音只是输入方式不同，不应导致关系状态或上下文重置。

#### 需求四：我需要知道角色现在怎么回应我，而不是盲聊

用户在聊天前常常会犹豫：

- 她现在还在生气吗？
- 他会不会不愿意理我？
- 现在适合说日常话题，还是应该回应刚才的剧情情绪？

聊天页需要给轻量上下文，而不是重文档说明。用户只需要一眼知道：

- 当前关系氛围是什么。
- 最近一次互动对这次对话会产生什么影响。
- 角色现在大概处于怎样的回应状态。

#### 需求五：我希望对话是“关系行为”，而不是一段孤立消息流

NaravoAI 的用户不只是要一个聊天记录容器，他们更在意：

- 这段对话是否会继续影响角色对我的看法。
- 角色是否会记住我刚才表达的内容。
- 这次对话是否可能改变我下一次剧情中的关系状态或开场氛围。

也就是说，聊天页必须让用户感受到：**我不是在给系统发消息，我是在继续经营一段关系。**

#### 需求六：我需要一个比剧情更低压、比详情页更直接的在场空间

剧情互动有成本，详情页偏信息组织，而聊天页的作用正好补足中间层：

- 我不想马上回剧情，但我也不想只看资料。
- 我不一定有很长时间，但我想和 TA 说一句话。
- 我可能只是想要一段几秒钟的回应，确认这段关系还在继续。

这个需求对于留存非常关键，因为很多复访不一定从“继续玩一整章”开始，而是从“先聊两句”开始。

## 7. 产品目标与非目标

### 7.1 产品目标

- 让用户从任一聊天入口进入后，在 3 秒内理解当前对话对象和关系上下文。
- 让用户在 5 秒内开始第一次输入，不被额外流程阻挡。
- 让用户在同一页面内自然切换文字输入和语音输入。
- 强化“角色持续存在、关系持续积累”的产品感知。
- 让聊天页成为剧情前预热、剧情后延续、日常维系三种场景的统一容器。
- 为后续关系成长、记忆管理、聊天回流推荐提供稳定页面载体。

### 7.2 非目标

- 本期不做群聊或多角色同时对话。
- 本期不做完整世界观设定阅读。
- 本期不做复杂对话模式选择器，例如“闲聊 / 剧情 / 心理陪伴 / RP”多模式并存。
- 本期不做可视化关系养成教学系统。
- 本期不把聊天页做成传统社交 IM 的完整工具集合，不需要文件、图片、表情商店等噪音能力。

## 8. 页面角色定义

角色聊天页在产品里应承担四个角色：

- 对话页：用户可以直接发送文本或语音。
- 关系承接页：显示当前关系与最近剧情结果对这次对话的影响。
- 陪伴页：允许用户不进剧情也能维持角色在场感。
- 转化页：在合适时机把用户带回具体 `Game / Chapter`。

它不是一个纯“发消息工具”，而是一个会随关系状态变化而变化的互动空间。

## 9. 典型使用场景

### 9.1 剧情前预热

用户看完角色详情页，知道最近关系不稳定，但还不想直接回到 `Chapter`。此时进入聊天页，先说一句：

> “你现在还在介意刚才那件事吗？”

这类场景的关键，是让用户以低风险方式确认角色当下状态。

### 9.2 剧情后延续

用户刚完成一个高情绪结尾，想趁余温继续和角色说话。此时聊天页承接的是情绪余波，而不是新会话。

### 9.3 日常陪伴

用户没有足够时间进入剧情，但想维持关系存在感，发一句文字或语音即可。

### 9.4 回流召回

用户一段时间没打开某个角色线，聊天页可以比“继续某章”更容易把人拉回来，因为聊天对时间和情绪成本都更低。

## 10. 信息架构

### 10.1 页面结构总览

建议页面从上到下分成 7 个层次：

1. 顶部导航区
2. 角色上下文头部
3. 关系提示条
4. 会话消息区
5. 输入增强区
6. 输入区
7. 底部剧情承接区

### 10.2 模块说明

#### 模块 A：顶部导航区

作用：让用户始终知道自己正在和谁聊天，并可安全返回。

建议包含：

- 返回按钮
- 角色头像
- 角色名
- 在线/状态文案，不建议用真实在线语义，建议用角色态文案
- More 菜单

More 菜单建议包含：

- 查看角色资料
- 查看关系摘要
- 查看最近关键事件
- 跳转继续剧情
- 静音角色语音播放（如后续支持）
- 举报内容

#### 模块 B：角色上下文头部

作用：在不打断聊天节奏的前提下，告诉用户“这次对话是从哪里接上来的”。

建议信息：

- 当前关系阶段
- 轻量关系状态标签，例如“信任回升中”“仍有防备”“高度依赖”
- 最近一次相关剧情摘要
- 当前对话起点说明

示例：

- 你们刚结束 `One Man Station · Ep. 5`
- 他现在愿意回应你，但情绪仍未完全稳定

这一层应该足够轻，不能像详情页模块那样重。

#### 模块 C：关系提示条

作用：给用户一个立即可理解的对话策略提示。

建议形式：

- 一句话提示当前推荐说话方式
- 可关闭或下滑后折叠

示例：

- 她现在更容易回应真诚确认，而不是追问细节。
- 他今天的状态适合短句交流和语音安抚。

这层不是“教程”，而是帮助用户降低开口成本。

#### 模块 D：会话消息区

作用：承载用户与角色的核心对话。

建议包含：

- 角色消息
- 用户消息
- 语音消息转写卡片
- 系统轻提示卡片
- 关系变化提示卡片

展示原则：

- 主体仍是自然对话。
- 关系变化提示不能过度频繁，避免像游戏弹数值。
- 系统提示要弱化存在感，更多服务理解而不是抢戏。

#### 模块 E：输入增强区

作用：降低开口成本，让用户不空白地面对输入框。

建议能力：

- 3 个以内建议开场
- 最近剧情相关的一句引导
- 一键发送的轻量 prompt

示例：

- 你还好吗？
- 我想继续刚才的话题。
- 我只是来看看你现在怎么样。

注意：建议语不应像模板问卷，必须足够人话。

#### 模块 F：输入区

作用：完成文字与语音输入。

建议包含：

- 文本输入框
- 发送按钮
- 语音按住说按钮
- 录音状态反馈
- 识别中 / 发送中状态

输入原则：

- 默认突出文本输入框，因为它理解成本最低。
- 语音按钮要足够明确，避免误触。
- 文字和语音走同一会话流，不出现“切换模式重开一轮”。

#### 模块 G：底部剧情承接区

作用：把聊天和剧情重新接起来。

建议触发方式：

- 默认弱展示，不抢输入区位置。
- 当系统判断“当前对话更适合回剧情继续”时再增强提示。

建议文案：

- 回到 `Kaleidoscope Heart · Ep. 2`
- 继续刚才的现场互动

## 11. 核心功能设计

### 11.1 入口收敛逻辑

所有入口进入同一个聊天页，但需要传入统一上下文对象：

- `characterId`
- `entrySource`
- `relationshipStage`
- `relationshipStatusLabel`
- `lastGameTitle`
- `lastChapterTitle`
- `lastInteractionSummary`
- `timelineContext`（可为空）

`entrySource` 建议至少区分：

- `home_cast_chat`
- `character_profile_hero_chat`
- `character_profile_quick_action_chat`

作用不是为了做三套页面，而是为了：

- 调整首屏文案。
- 调整默认提示强度。
- 方便后续埋点分析。

### 11.2 会话上下文继承

聊天页必须显式继承角色上下文，而不是只传一个角色 ID。

至少需要继承：

- 当前角色身份
- 当前关系阶段
- 最近一次剧情互动结果
- 最近一次关系波动摘要

如果技术条件允许，建议再补：

- 最近 1 到 3 个关键关系节点摘要
- 最近一次用户对该角色的主动表达

### 11.3 文本聊天

文本聊天是首期最基础也最稳定的能力。

设计要求：

- 支持多轮连续发送。
- 发送后立即进入角色思考 / 生成反馈态。
- 角色回复可分段显示，但不要过慢拖节奏。
- 对于明显承接剧情的问题，角色回复要体现上下文记忆。

### 11.4 语音聊天

语音聊天应作为正式能力设计，而不是附属按钮。

首期建议交互：

- 按住说话开始录音。
- 松开发送。
- 识别结果以转写消息形式进入会话流。
- 角色回复仍以文本为主；如后续接 TTS，可再扩展为角色语音播放。

这样做有几个好处：

- 不会让首期复杂度暴涨到实时双工语音通话。
- 仍然保留“我可以直接开口说话”的强体验。
- 和当前项目中已有的 ASR / TTS 能力路径更接近。

### 11.5 语音输入反馈

语音输入必须有明确反馈，否则用户会不安：

- 录音中：显示波形或呼吸灯态
- 识别中：显示短暂处理中状态
- 识别完成：以转写卡片进入消息区
- 失败时：允许“一键重试”或“转为文字发送”

### 11.6 关系提示与变化反馈

聊天页不建议把关系数值长期悬浮展示，但应该让用户感受到关系有反馈。

建议使用两层机制：

1. 静态层

- 顶部显示当前关系阶段与状态标签

2. 动态层

- 当出现明显关系变化时，在消息流中插入轻量提示

示例：

- 她对你的戒备稍微下降了。
- 他开始更愿意主动解释自己的处境。

注意，这类提示要像关系观察，而不是像数值结算页。

### 11.7 建议开场与空状态承接

对于第一次聊天或低互动角色，用户最容易卡住。

因此需要提供空状态承接：

- 角色首句欢迎
- 与当前身份相关的一句提示
- 2 到 3 个建议开场

新角色空状态重点：

- 帮用户开始第一句。

老角色空状态重点：

- 帮用户接上当前关系。

### 11.8 会话中的剧情回跳

聊天页不应把用户困在对话里。

在以下时机可以给出“回到剧情”的轻提示：

- 用户明确提到刚才剧情中的事件
- 角色回复中自然提到“这件事要回现场处理”
- 当前故事本来就停在高紧张节点

但这类提示应该是引导，不是强制中断。

### 11.9 历史会话与长期连续性

首期建议保留同角色的持续聊天线程，不要每次进入都开一个新会话。

原因：

- 这更符合“角色持续存在”的认知。
- 用户更容易形成习惯性回访。
- 角色的长期记忆更容易被用户体感到。

但需要解决一个问题：

- 长期线程会越来越长，不利于回看。

因此首期虽然不一定做复杂历史管理，也建议预留：

- 按时间折叠旧消息
- 标记关键剧情切入点
- 重要记忆卡片锚点

## 12. 交互设计

### 12.1 首次进入交互

用户第一次从入口进入聊天页时，建议顺序如下：

1. 页面打开，顶部显示角色与关系状态。
2. 头部显示一条轻量上下文提示。
3. 若存在最近剧情承接，展示一条系统提示卡。
4. 输入区自动就绪，用户可直接打字或按住说话。

目标不是“展示很多东西”，而是“让用户立刻愿意说第一句话”。

### 12.2 从 Home 的 cast 入口进入

建议承接方式：

- 首屏更轻，不占太多垂直空间。
- 默认突出“现在开始聊”。
- 如果该角色最近有强剧情上下文，只展示一条摘要，不要做重型介绍。

因为从 `Home` 进入的用户，更像是来快速挑一个想聊的人。

### 12.3 从 character profile 进入

建议承接方式：

- 顶部上下文更强。
- 强调当前关系状态与最近事件。
- 如果用户是从 Hero 区进入，可默认让页面更靠近输入区。
- 如果用户是从 Quick Actions 或底部 CTA 进入，可在开场多保留一条上下文提示。

因为这类用户刚完成对关系的理解，聊天页需要顺势接住。

### 12.4 输入交互

文本输入建议：

- 支持多行，但高度不要过度扩张。
- 当用户开始输入时，建议开场自动收起。
- 发送按钮只有在有内容时高亮。

语音输入建议：

- 采用按住说话，避免误录。
- 按住后给明显视觉反馈。
- 松手即提交，向上滑取消可作为后续增强项。

### 12.5 角色回复交互

角色回复建议：

- 优先保证节奏顺畅，避免长时间沉默。
- 若角色回复较长，可分段出现，但总时延要克制。
- 如果后续支持 TTS，文字应先可读，语音播放作为增强，而不是阻塞阅读。

### 12.6 系统提示交互

系统提示在聊天页只能做辅助角色。

适合出现的内容：

- 关系承接说明
- 轻量关系变化
- 回剧情建议

不适合出现的内容：

- 大段教学说明
- 频繁的数值变化弹窗
- 太强的系统感文案

### 12.7 返回与页面退出

返回逻辑建议：

- 返回到原入口页面
- 保留原页面滚动位置
- 若用户有未发送文本，提示是否保留草稿

原因：

- `Home` 用户可能想继续在 cast 列表里切换角色。
- `character profile` 用户可能想继续回看关系模块或改走剧情入口。

### 12.8 异常与中断状态

必须设计以下状态：

1. 网络异常

- 用户消息保留
- 支持重新发送

2. 语音权限被拒绝

- 明确提示原因
- 提供“改用文字输入”

3. 识别失败

- 展示失败原因
- 提供重新录音

4. 角色回复超时

- 显示处理中
- 超时后提供重试或刷新上下文

5. 无最近剧情上下文

- 仍可正常聊天
- 页面退化为角色基础关系对话态

## 13. 页面布局建议

### 13.1 移动端布局

建议采用典型单列结构：

1. Sticky 顶部导航
2. 轻量上下文头部
3. 可滚动消息流
4. 底部输入区
5. 弱化的剧情承接入口

优先级排序：

- 第一优先级：消息流与输入区
- 第二优先级：关系上下文
- 第三优先级：回剧情承接

### 13.2 低保真结构示意

```text

```

## 14. 文案与语气建议

聊天页的文案必须避免太像工具页或客服页。

建议原则：

- 强调“关系”而不是“功能”。
- 强调“承接”而不是“重开”。
- 强调“在场”而不是“在线”。

建议用法：

- 与 TA 聊天
- 现在和 TA 说句话
- 以当前关系状态开始对话
- 你们刚刚结束在 `XXX` 的互动

不建议用法：

- 开启会话
- 启动聊天模式
- 进入语音模式
- 在线中

因为这些说法太工具化，容易破坏角色氛围。

## 15. 关键设计原则

### 15.1 不把它做成普通聊天工具

用户来这里，不是为了一个万能 IM。

### 15.2 不把它做成角色详情页的复制品

详情页负责解释关系，聊天页负责继续关系。

### 15.3 不让语音成为花哨演示

语音应该降低表达门槛，而不是抬高操作学习成本。

### 15.4 不让上下文说明压过聊天本身

上下文是帮助用户开口，不是让用户先读文档。

## 16. 首期 MVP 建议

### 16.1 必做

- 统一聊天页路由/容器
- 支持 3 个入口进入同一页面
- 顶部角色信息与轻量关系上下文
- 文本输入与多轮消息流
- 按住说话 -> ASR -> 转写入流
- 角色回复承接当前关系与最近剧情
- 基础异常状态处理
- 回到最近 `Game / Chapter` 的剧情入口

### 16.2 可选增强

- 建议开场
- 关系变化轻提示
- 草稿保留
- 语音转写重编辑

### 16.3 暂缓

- 实时双工语音通话
- 完整历史检索
- 多角色房间
- 图片/附件发送
- 高复杂度人格模式切换

## 17. 关键指标建议

建议至少跟踪以下指标：

- 聊天页入口点击率
- 各入口到首条消息发送转化率
- 文本输入占比 / 语音输入占比
- 首条消息发送耗时
- 聊天页到剧情页回跳率
- 单角色聊天复访率
- 单次会话轮数
- 语音识别失败率

重点观察问题：

- 用户是否真的把聊天页当作“关系延续空间”使用。
- 哪个入口进入后最容易开始首轮对话。
- 语音功能是在增益体验，还是增加流失。

## 18. 风险与注意事项

### 18.1 风险一：做成普通聊天页，NaravoAI 差异消失

如果页面只剩头像、消息和输入框，它会迅速滑向 generic AI chat。

### 18.2 风险二：上下文太重，用户开不了口

如果页面一进来堆太多关系信息、剧情说明、系统文案，用户会从“我想说句话”变成“我要先读完说明”。

### 18.3 风险三：语音体验不稳定，反而损伤信任

语音能力一旦识别差、反馈慢、状态不明，用户会更容易退出。

### 18.4 风险四：聊天和剧情完全断裂

如果聊天不能影响剧情感受，或者角色明显不记得最近事件，用户会迅速认为这是两套割裂系统。

## 19. 后续可扩展方向

- 角色语音回复播放
- 记忆卡片沉淀与可回看
- 聊天中触发剧情分支推荐
- 关系节奏提醒，例如“适合回去推进这一章了”
- 基于关系状态生成更自然的开场建议
- 更精细的入口场景化承接

## 20. 一句话结论

角色聊天页不是给 NaravoAI 补一个“也能聊天”的功能，而是把“剧情内建立的关系”延长到剧情之外，让角色真正从一段内容，变成一个持续存在、可被再次抵达的人。
