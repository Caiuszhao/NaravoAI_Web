# 角色详情页 PRD

## 1. 文档信息

- 产品名称：NaravoAI Website
- 模块名称：Characters Tab -> Character Detail Page
- 文档版本：v0.1
- 更新时间：2026-04-26
- 文档性质：产品需求与交互设计草案

## 2. 背景

NaravoAI 不是一个单纯看剧情的视频产品，也不是一个只做陪聊的 AI 角色产品。它更像是把互动叙事、角色关系、持续记忆放在同一个体验里：用户在具体 Game 和 Chapter 中与角色互动，这些互动会累积成关系，并反过来影响用户下一次进入故事时的心理预期和行动选择。

现在 `Characters` tab 已经能让用户快速浏览“最近互动过的角色卡片”，但它只回答了两个很浅的问题：

- 我最近和谁互动过。
- 当前大概是什么关系状态。

它还没有回答更重要的问题：

- 这个角色对我来说到底意味着什么。
- 我们之间发生过哪些关键事件。
- 我和她/他是在什么故事里建立起现在这层关系的。
- 如果我现在想继续推进关系，下一步应该去哪里。

这就是角色详情页存在的理由。它不是“角色百科”，也不是“纯聊天记录页”，而是一个面向用户的关系总览页，帮助用户快速理解自己和某个角色之间的关系全貌，并自然进入下一次互动。

## 3. 产品定位

角色详情页的定位可以定义为：

**一个面向用户的“角色关系主页”，负责把分散在不同 Game、不同 Chapter、不同互动形式里的信息，整理成可理解、可回顾、可继续推进的关系视图。**

它承接三层价值：

- 对用户：降低记忆负担，让关系更有连续感。
- 对角色：把“有记忆、会成长”的设定具体展示出来。
- 对产品：把一次次短互动，沉淀成长期留存和复访动机。

## 4. 市场参照与机会判断

从市场上相近的两类产品看，用户的预期已经比较清晰：

### 4.1 AI companion 类产品给用户建立的预期

这类产品通常强调：

- 角色会记住我。
- 关系会随着互动推进。
- 角色有稳定的人设和情感反馈。

它们的优势是“关系连续性”强，但弱点也很明显：

- 用户常常看不到关系是如何被塑造出来的，只能在聊天里体感变化。
- 历史互动缺少结构化整理，久了以后会变成一段很长但不好回看的对话。
- 跨故事、跨场景的信息组织不够清楚。

### 4.2 互动叙事 / 互动恋爱类产品给用户建立的预期

这类产品通常强调：

- 我在哪个故事里认识了这个角色。
- 我推进到了哪一章。
- 我的选择影响了剧情和关系。

它们的优势是“故事进度”清楚，但也有不足：

- 角色关系常常只在单个故事内成立，出了这条线就不够连续。
- 用户能看到章节进度，却不一定能看到“关系变化的原因和结果”。
- 角色页更像档案页，而不是持续关系页。

### 4.3 NaravoAI 的机会

NaravoAI 的差异点不在于只做“有剧情的角色”，而在于把下面三件事放在一起：

- 互动发生在具体故事里。
- 角色关系会持续积累。
- 用户下一次进入故事时，会带着过去的情绪和关系状态继续玩。

因此，角色详情页要补上市场上常见产品的断层：

- 既要像 companion 产品那样让用户感到“这段关系是真的在持续”。
- 也要像互动叙事产品那样让用户明确知道“这段关系是在哪些内容里被建立起来的”。

## 5. 目标用户需求

### 5.1 用户的核心任务

当用户点开角色卡片时，通常不是为了看一份静态资料，而是为了完成下面几种任务：

1. 快速确认我和这个角色现在是什么关系。
2. 回想我们之前经历过什么，为什么会走到今天这一步。
3. 确认我和这个角色相关的内容还剩什么没玩、该从哪里继续。
4. 判断这个角色值不值得我继续投入时间和情绪。
5. 感受到这个角色对我有持续记忆，而不是一次性消耗内容。

### 5.2 角色详情页满足的用户需求

#### 需求一：关系要“看得懂”

用户需要一眼看懂：

- 当前关系身份，例如陌生人、熟人、朋友、闺蜜、恋人、战友、暧昧对象。
- 当前关系温度，例如信任上升、关系不稳、依赖增强、情感冻结。
- 当前关键指标，例如信任度、依赖度、亲密度、默契度、风险值。

如果没有这些明确表达，用户会知道“好像变亲近了”，但说不清到底变了什么。

#### 需求二：历史要“回得去”

用户需要知道：

- 我和这个角色第一次在哪个 Game 遇见。
- 哪些 Chapter 对关系影响最大。
- 最近一次互动发生在哪个 Chapter。
- 最近一次互动的情绪结果是什么。

如果只有最后一句台词和一个时间标签，用户很难把零散互动拼成完整关系。

#### 需求三：下一步要“接得上”

用户打开详情页后，往往已经处在犹豫状态：继续玩谁、继续推哪条线、要不要重新进入某个故事。

所以页面必须给出明确下一步：

- 继续上次 Chapter
- 重温关键互动
- 查看该角色相关的其他 Game
- 查看待解锁内容

如果详情页只能“看”，不能“接着玩”，它就只是信息页，不是转化页。

#### 需求四：角色要“像一个连续存在的人”

NaravoAI 的角色不是卡牌，不是静态立绘，也不是一次性剧情 NPC。用户需要感受到：

- 这个角色有连续存在感。
- 产品记得我和她/他共同发生过什么。
- 关系状态不是系统凭空给的，而是由互动积累出来的。

这决定了详情页不能只是堆字段，而要把“关系为什么是这样”讲清楚。

#### 需求六：我想暂时不回剧情，但还想和角色继续说话

并不是每个用户打开详情页时都准备立刻进入某个 Game 或 Chapter。很多时候，用户只是想：

- 先和角色说一句话，确认当下关系氛围。
- 在进入高情绪剧情前，先试探角色现在的回应状态。
- 暂时不推进主线，只维持陪伴感和在场感。

如果详情页只有“继续剧情”而没有“进入聊天”，页面会把用户的下一步选择压缩得太窄。对于 NaravoAI 这类同时强调角色持续性和互动关系的产品来说，聊天入口应该是角色详情页里的一个正式能力，而不是附加功能。

#### 需求五：内容边界要“讲清楚”

如果一个角色分布在多个 Game、多个分支、多个 Chapter 中，用户很容易混乱：

- 哪些内容我玩过。
- 哪些内容和这个角色强相关。
- 哪些是主线，哪些是补充内容。

角色详情页需要承担整理和导航作用，帮助用户建立稳定的内容认知。

## 6. 产品目标与非目标

### 6.1 产品目标

- 让用户在 10 秒内理解自己与该角色的当前关系。
- 让用户在 20 秒内回顾和该角色相关的互动脉络。
- 让用户在当前页直接找到最合适的下一步互动入口。
- 让用户在“继续剧情”和“直接聊天”之间快速做出选择。
- 强化“角色有记忆、关系会积累”的产品感知。
- 为后续做关系成长、内容推荐、回流召回提供稳定页面容器。

### 6.2 非目标

- 本期不做完整聊天记录逐条展开阅读。
- 本期不做复杂关系养成系统的数值教学。
- 本期不在详情页内直接承载完整互动流程，互动仍回到具体 Game / Chapter。
- 本期不把它做成世界观百科页，设定信息只展示与关系理解强相关的部分。

## 7. 页面角色定义

角色详情页可以视为一个三合一页面：

- 关系总览页：告诉用户“现在是什么关系”。
- 互动回顾页：告诉用户“之前发生了什么”。
- 行动导航页：告诉用户“现在可以去做什么”。
- 对话进入页：告诉用户“如果不回剧情，现在也可以直接和角色说话”。

页面不该平均分配篇幅，而应该有主次：

- 第一优先级：关系状态与下一步行动。
- 第二优先级：互动历史与关键节点。
- 第三优先级：角色资料与扩展内容。

这里的“下一步行动”要明确包含两类入口：

- 回到具体 Game / Chapter 继续剧情互动。
- 进入与角色的一对一聊天，延续关系但不强制推进剧情。

## 8. 信息架构

### 8.1 页面结构总览

建议从上到下分成 8 个模块：

1. 顶部概览 Hero
2. 关系状态模块
3. 快速行动模块
4. 互动历史总览模块
5. 关键事件时间线模块
6. 相关 Game / Chapter 模块
7. 角色资料模块
8. 底部行动区

### 8.2 模块说明

#### 模块 A：顶部概览 Hero

作用：让用户一进来就知道“这是哪个角色、我现在和她/他是什么状态、最近发生了什么”。

建议信息：

- 角色大图 / 动态头像
- 角色名
- 角色一句话身份标签
- 当前关系称谓
- 当前状态标签，例如“信任回升中”“关系不稳定”“高度依赖”“情感确认中”
- 最近一次互动摘要
- 主按钮：继续互动
- 次按钮：进入聊天
- 辅助文字入口：查看互动历史

这一屏承担的是“立即理解”和“立即行动”。

Hero 区的两个主行动建议分工明确：
- `继续互动`：回到最近一次相关的 Game / Chapter。
- `与 TA 聊天`：进入角色聊天页，不强制承接某个 Chapter。

#### 模块 B：关系状态模块

作用：把抽象关系变成可感知信息。

建议信息：

- 关系阶段：陌生人 / 熟人 / 朋友 / 挚友 / 闺蜜 / 恋人 / 战友 / 搭档
- 信任度
- 依赖度
- 亲密度
- 默契度
- 当前波动提醒，例如“最近两次互动让她对你产生犹豫”

设计建议：

- 不建议只展示冷冰冰进度条。
- 每个指标最好同时有简短解释，例如：
  - 信任度 78：她更愿意向你暴露真实情绪。
  - 依赖度 62：她在压力情境下开始优先寻求你的回应。

这样用户不会只看到数值，还能理解数值背后的情感含义。

#### 模块 C：快速行动模块
作用：解决“我现在该做什么”的问题，并把两个高频动作直接摆在用户面前。

建议放在关系状态模块之后、历史模块之前，因为用户在理解完当前关系后，往往立刻就要做决定。

建议包含两个主入口：
- 继续上次剧情
- 进入与角色聊天

建议同时附上简短说明：
- 继续上次剧情：回到 `Game + Chapter`，沿着当前故事继续推进。
- 进入与角色聊天：不进入剧情现场，直接以当前关系状态开启对话。

如果产品后续支持，还可以在此模块展示一个轻量提示：
- 角色当前对话状态，例如“她现在愿意回应你，但情绪仍有防备”

#### 模块 D：互动历史总览模块
作用：帮助用户快速建立“关系地图”。

建议信息：
- 累计互动次数
- 涉及 Game 数量
- 已体验 Chapter 数量
- 最近一次互动时间
- 第一次相遇的 Game / Chapter
- 最常互动模式，例如 Voice / Text / Tap / Chat

该模块不需要太长，但要能让用户快速对这段关系形成尺度感。

#### 模块 E：关键事件时间线模块
作用：告诉用户这段关系是怎么演变的。

建议展示 3 到 7 个关键节点，优先展示：

- 初次相遇
- 第一次关系升级
- 重要分歧
- 最近一次高强度互动
- 关键解锁点

每个节点建议包含：

- 时间或相对时间
- Game 名称
- Chapter 名称
- 事件标题
- 一句结果说明
- 状态变化，例如“信任 +12 / 依赖 +8 / 关系从朋友进入暧昧”
- 可点击进入“重温该节点”

这部分是页面里最能体现 NaravoAI 产品差异的内容，因为它把“角色记忆”从抽象设定变成有证据的关系演变。

#### 模块 F：相关 Game / Chapter 模块
作用：解决“接下来去哪里”的问题。

建议拆成两个区：

1. 继续当前关系线

- Continue last chapter
- 当前所在 Game
- 最近进度
- 预计继续时长

1. 该角色相关的其他内容

- 已解锁但未开始的相关 Game
- 该角色高参与度的其他剧情
- 支线 / 隐藏线 / 特殊互动入口

排序逻辑建议：

- 第一位永远是“继续上次互动”
- 第二位是“最能推进关系的下一内容”
- 后面才是探索型内容

#### 模块 G：角色资料模块
作用：补足认知，但不喧宾夺主。

建议信息：

- 角色身份 / 职业 / 阵营
- 性格关键词
- 关系偏好或触发点
- 用户已知设定与未知设定
- 角色相关世界观标签

这里重点不是做百科，而是服务互动：

- 什么会让她更信任你
- 什么会触发她的防御
- 她通常在什么情境下依赖你

#### 模块 H：底部行动区

作用：把回顾转成下一次互动。

建议按钮：

- 与角色聊天
- 继续上次互动
- 重温关键章节
- 查看相关 Game

底部行动区要保持始终明确，不能让用户看完后不知道下一步去哪里。

## 9. 建议的页面布局

### 9.1 移动端布局建议

移动端建议采用单列纵向滚动，顺序如下：

1. 顶部导航
2. Hero 区
3. 关系状态卡

- 用户从角色卡进入后，第一关注点一定是“我们现在是什么状态”。

## 10. 核心功能设计

### 10.1 入口设计

当前入口为 `Characters` tab 中的角色卡片。

建议交互：

- 点击卡片任意区域进入详情页。
- 保留卡片到详情页的转场关联感，例如头像放大、背景延展、关系标签跟随进入。
- 返回时回到原来的卡片列表位置，避免用户丢失浏览上下文。

如果角色卡片后续需要支持快捷操作，建议长按或右上角更多菜单再提供“直接聊天”，不要在卡片首层堆太多按钮，避免列表页失焦。

### 10.2 顶部导航

建议包含：

- 返回按钮
- 页面标题或角色名
- More 菜单

More 菜单可放：

- 分享角色
- 隐藏该角色
- 举报内容
- 查看完整设定
- 发起聊天

### 10.3 关系状态可视化

建议采用“标签 + 指标 + 一句解释”的组合，而不是只放多个进度条。

原因：

- 纯进度条很像游戏数值页，情感感弱。
- 只放文字又会缺少可比较性。

推荐形式：

- 关系阶段用大标签表示
- 4 个关键指标用紧凑卡片表示
- 下方加一行系统解释文案

示例：

- 当前关系：朋友以上，尚未确认
- 信任度：78
- 依赖度：62
- 亲密度：71
- 当前状态：她开始主动向你暴露脆弱面，但仍害怕被拒绝

### 10.4 互动历史呈现

不建议直接展示完整聊天记录，信息太重。

建议分两层：

- 第一层：概览卡片，给出总量和关键节点。
- 第二层：时间线节点，按重要性展示。

如果后续需要扩展，再加入“查看完整互动记录”。

### 10.5 聊天入口设计

聊天入口建议作为角色详情页里的核心入口之一，地位仅次于“继续上次剧情”。

建议放置在三个位置：

1. Hero 区次按钮
- 文案建议：`与 TA 聊天`
- 适合用户刚进入页面时的即时决策

2. 快速行动模块
- 和“继续上次剧情”并列
- 用于用户在理解当前关系后进行选择

3. 底部 CTA
- 作为浏览完整页后的稳定兜底入口

聊天入口的设计目标不是替代剧情入口，而是承接另一种真实需求：用户此刻想维持关系、确认情绪、试探回应，但不一定想立刻回到具体 Chapter。

建议入口文案避免过于工具化，优先考虑：
- 与 TA 聊天
- 现在聊聊
- 发送消息

不建议只写成：
- Chat

因为用户需要在产品语境里立刻明白，这是“进入角色关系对话”，不是一个普通客服或系统消息入口。

### 10.6 聊天入口与剧情入口的关系

这两个入口必须被清楚区分，否则用户会不知道点哪个。

建议区分逻辑如下：

- `继续上次剧情`
  - 目标：推进叙事进度
  - 承接对象：具体 Game / Chapter
  - 用户心态：我准备继续玩

- `与 TA 聊天`
  - 目标：延续关系、确认角色当下状态、进行轻量互动
  - 承接对象：角色私聊 / 角色对话空间
  - 用户心态：我现在更想先和她/他说话

建议在按钮下方加一句轻提示，帮助用户做区分：
- 继续上次剧情：回到 `Kaleidoscope Heart · Ep. 2`
- 与 TA 聊天：以当前关系状态开始对话

### 10.7 Chapter 与 Game 导航

这是详情页最关键的转化能力之一。

建议把内容分成三种状态：

- Continue：最近玩到这里，继续即可。
- Replay：这是关键节点，适合回顾。
- Discover：这是相关内容，但你还没体验。

用户一眼就能明白每个入口的用途，而不是只看到一堆内容标题。

### 10.8 角色资料的呈现边界

资料模块建议以“与互动强相关的信息”为主：

- 她如何建立信任
- 她会对什么反应敏感
- 她在哪些剧情里最有代表性

不建议首期放太多纯设定资料，否则会稀释详情页的关系主题。

## 11. 交互设计

### 11.1 用户路径

建议用户路径如下：

1. 用户在 `Characters` tab 浏览角色卡片。
2. 用户点击某张角色卡。
3. 进入角色详情页，默认停留在顶部总览。
4. 用户先确认当前关系状态。
5. 用户在“继续剧情”或“进入聊天”之间做选择。
6. 用户向下浏览关键事件和相关内容。
7. 用户通过主 CTA 回到具体 Game / Chapter，或进入与角色聊天。

这条路径的关键不是“停留更久”，而是“更快做出继续互动的决定”。

### 11.2 Hero 区交互

建议支持：

- 点击主按钮直接继续最近一次互动
- 点击次按钮进入角色聊天页
- 点击最近一次互动摘要，跳到时间线中的对应事件
- 角色视频封面可轻微动态播放，增强角色存在感

### 11.3 时间线交互

建议每个关键节点支持：

- 展开查看事件摘要
- 跳转到对应 Game / Chapter
- 标记关系变化

建议默认只展示最重要的 3 个节点，点击后可展开完整时间线。这样不会让首屏过重。

### 11.4 相关内容交互

建议每张内容卡都明确标记：

- 所属 Game
- 当前 Chapter
- 与角色的相关性
- 当前状态：继续 / 已完成 / 未解锁

避免用户看见内容列表，却不知道为什么它会出现在这个角色页里。

### 11.5 聊天入口交互

建议聊天入口点击后直接进入独立聊天页或聊天浮层，进入后需要继承当前角色上下文，包括：

- 当前角色身份
- 当前关系阶段
- 最近一次关系状态
- 可选的最近互动摘要

如果技术上允许，建议在聊天页顶部给出一条轻提示，帮助用户理解这次对话和关系上下文的连接，例如：
- 你们刚刚结束在 `One Man Station · Ep. 5` 的互动
- 她现在对你保持高信任，但情绪仍不稳定

这样聊天不会显得像一段和剧情割裂的“新会话”。

### 11.6 返回交互

从详情页返回时，应回到 `Characters` tab 的原位置。

原因：

- 用户可能会连续查看多个角色。
- 如果返回后总是回到顶部，浏览成本会明显上升。

### 11.7 空状态与异常状态

必须设计以下状态：

1. 新角色 / 低互动状态

- 没有完整历史时，重点展示角色基础信息和“开始第一次互动”。

1. 单一 Game 角色

- 没有“其他相关 Game”时，改为展示“可重温章节”或“即将解锁内容”。

1. 高互动角色

- 历史很多时，只显示关键节点和折叠时间线。

1. 数据尚未完善

- 某些指标没有计算结果时，显示“关系仍在形成中”，而不是空白。

## 12. 页面内容优先级

## 15. 设计原则

- 一眼看懂当前关系，不让用户自己推理。
- 所有历史信息都要服务“现在关系”和“下一步行动”。
- 页面要有情感密度，但不能做成一页复杂报表。
- 数据表达要可解释，不能只有数字没有含义。
- 每个模块都应该回答用户心里的一个具体问题。

## 16. 成功指标

上线后可以重点看以下指标：

- `Characters` tab 卡片点击率
- 角色详情页进入率
- 详情页到具体 Game / Chapter 的跳转率
- 详情页到聊天页的点击率
- 详情页进入后的继续互动转化率
- 详情页进入后的聊天发起率
- 角色详情页停留时长
- 查看详情页后的 7 日角色复访率

更关键的是看两个行为变化：

- 用户是否更愿意围绕“某个角色”持续回来，而不是只围绕“某个 Game”回来。
- 用户是否更容易理解关系为什么变化，从而更愿意继续投入。

## 17. 风险与注意事项

### 17.1 过度数值化风险

如果信任度、依赖度、亲密度做得太像养成游戏数值面板，会破坏角色真实感。

解决建议：

- 数值必须搭配自然语言解释。
- 把“状态”放在“数值”之前。

### 17.2 信息堆积风险

如果把所有历史都塞进详情页，页面会很重。

解决建议：

- 首屏只给最重要信息。
- 历史按“关键节点”组织，不按“全部记录”组织。

### 17.3 用户误解风险

如果关系状态和推荐内容之间没有清晰因果，用户会觉得系统在随便打标签。

解决建议：

- 每个重要状态都要能追溯到具体事件。
- 每个推荐入口都要说明“为什么推荐给你”。

### 17.4 入口竞争风险

如果“继续剧情”和“与 TA 聊天”两个入口视觉权重接近，但分工不清，用户容易犹豫，反而降低转化。

解决建议：

- 在文案上明确两个入口的差异。
- Hero 区保留一主一次，不建议做两个完全同权重的主按钮。
- 在快速行动区补充解释文案，而不是只放按钮。

## 18. 一句话结论

这个角色详情页本质上不是一个资料页，而是 NaravoAI 的“关系中枢页”。它要把角色记忆、互动历史、关系状态、聊天入口和下一步行动连成一条线，让用户不只是看见一个角色，而是看见自己和这个角色正在形成并持续推进的关系。
